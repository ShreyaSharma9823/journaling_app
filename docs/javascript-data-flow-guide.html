JavaScript Data Flow Guide

For the journal app in this project

Big Picture

The JavaScript in templates/page.html connects the browser UI, the Django API, and the database. The browser does not save journals directly. It only collects values from the page and sends them to Django. Django validates the data and saves it through the serializer and model.

Save flow:

  1. User types title, date, and content.
  2. saveData() reads those values from the page.
  3. toIsoStringFromLocalInput() converts the date into a standard format.
  4. JSON.stringify() converts the JavaScript object into JSON text.
  5. fetch('/api/journals/', ...) sends it to Django.
  6. The serializer parses the datetime.
  7. The model saves the row in the database.

Read flow:

  1. loadJournals() sends a GET request.
  2. Django returns only the logged-in user's journals.
  3. renderJournalList() builds the sidebar UI.
  4. Clicking an item loads its data back into the editor.

Why JSON Is Used

JSON is not the database. JSON is just the format used to send structured data between the browser and Django over HTTP.

{
  "journal_title": "My day",
  "journal_description": "<p>Hello</p>",
  "journal_date": "2026-04-08T09:30:00.000Z"
}

So the rule is simple:

  • JavaScript creates the object
  • JSON carries it over the network
  • Django saves it in models

JavaScript Functions

getCookie(name)

Why: Django requires a CSRF token for POST and DELETE requests.

How: It reads document.cookie, finds csrftoken, and sends it in request headers.

toDateTimeLocalValue(dateValue)

Why: The API returns an ISO datetime, but datetime-local expects a browser-style local value.

How: It creates a Date, adjusts for timezone offset, and returns YYYY-MM-DDTHH:mm.

toIsoStringFromLocalInput(dateValue)

Why: The browser input is local time, while the API should receive a clean ISO timestamp. If the field is blank, current time should be used.

How: It returns new Date().toISOString() when blank, otherwise converts the chosen local time into ISO UTC text.

resetEditor()

Why: Needed when creating a fresh journal.

How: Clears the title, clears the date, clears the editor, and removes the selected journal id.

renderJournalList(journals)

Why: API data must become clickable UI.

How: Builds sidebar items, adds click events, and loads selected journal values back into the form.

loadJournals(searchValue = '')

Why: Loads journals on page open and refreshes them after save or delete.

How: Builds the API URL, calls fetch(), parses JSON with response.json(), and passes the result to renderJournalList().

deleteJournal(journalId)

Why: Removes a saved journal.

How: Confirms with the user, sends a DELETE request, and reloads the list.

saveData()

Why: This is the main save function.

How: Reads values from the DOM, validates title, builds a JavaScript object, converts it with JSON.stringify(), and sends it to Django with fetch().

Authentication In Your Project

Authentication means: who is the user?

Django handles this in dreamproject/views.py using:

  • User.objects.create_user(...) for signup
  • authenticate(...) to verify credentials
  • login(request, user) to create a session
  • logout(request) to end the session

After login, Django stores the session and gives you request.user on later requests.

Authorization In Your Project

Authorization means: what is this user allowed to do?

You are enforcing that in two places:

  • @login_required(login_url='index') protects the journal page.
  • permission_classes = [IsAuthenticated] protects the journal API.

The ownership rule is:

Journal_uploaded.objects.filter(owner=self.request.user)

That means each user only sees their own journals.

And on create:

serializer.save(owner=self.request.user)

That means the logged-in user becomes the owner automatically, which is the correct approach.

CSRF Protection

Your frontend reads the CSRF token from cookies and sends it as X-CSRFToken for save and delete requests. Django's CSRF middleware checks that token before allowing the action.
